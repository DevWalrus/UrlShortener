# infra

All Azure infrastructure for clinten.dev, managed by Terraform. The only resource not managed here is the Blob Storage account used for Terraform's own remote state — that must be created once manually before Terraform can run.

## Resources

| File | What it manages |
|---|---|
| `main.tf` | Provider config (azurerm, azuread) |
| `backend.tf` | Remote state in Azure Blob Storage (`clintentfstate`) |
| `resource_group.tf` | `rg-clinten` resource group |
| `cosmosdb.tf` | Cosmos DB account (free tier), `clintendev` and `clintendev-stage` databases, `links`/`users` collections with indexes |
| `container_registry.tf` | Azure Container Registry (`clintenregistry`) |
| `keyvault.tf` | Key Vault, managed identity access policies, secrets for MongoDB URI / ACR password / API key |
| `container_apps.tf` | Log Analytics workspace, Container App environment, managed identities, `clinten-redirect`, `clinten-creator-api`, `clinten-creator-api-stage` |
| `static_web_app.tf` | `creator-ui` (prod) and `creator-ui-stage` Static Web Apps with app settings |
| `aad.tf` | GitHub Actions service principal, OIDC federated credentials for `prod` and `stage` environments |
| `outputs.tf` | Hostnames, FQDNs, client/tenant IDs, Key Vault URI |

## Secret flow

Most secrets are generated by Azure when resources are created and read directly by Terraform — no manual handling required:

```
Azure creates resource (Cosmos DB, ACR, etc.)
        │  secret is Azure-generated, never typed by a human
        ▼
Terraform reads it via resource attribute
  e.g. azurerm_cosmosdb_account.main.primary_mongodb_connection_string
       azurerm_container_registry.main.admin_password
        │
        ▼
Terraform writes it to Key Vault
        │
        ▼
Container App secret block (references Key Vault secret URI)
        │
        ▼
Container environment variable (MONGODB_URI, API_KEY, etc.)
        │
        ▼
os.Getenv() in Go / process.env in Node
```

`variables.tf` contains only non-secret configuration (`location`, `project`, `subscription_id`) — there are no secrets in `terraform.tfvars`. The secrets that must be set manually in GitHub (see below) are identifiers or deployment tokens, not credentials.

## One-time bootstrap

Create the storage account for Terraform remote state before running `terraform init`:

```bash
az login

az group create --name rg-clinten-tfstate --location eastus2

az storage account create \
  --name clintentfstate \
  --resource-group rg-clinten-tfstate \
  --sku Standard_LRS

az storage container create \
  --name tfstate \
  --account-name clintentfstate
```

## First-time apply

```bash
cd infra
terraform init
terraform plan
terraform apply
```

After apply, bind custom domains in the Azure portal:
- `clinten.dev` → `clinten-redirect` Container App → Custom Domains
- `create.clinten.dev` → `creator-ui` Static Web App → Custom Domains

Add DNS records in Porkbun:

| Type | Host | Value |
|---|---|---|
| A | `@` | Container App environment static IP |
| CNAME | `create` | SWA default hostname |
| TXT | `asuid` | Redirect app domain verification ID |

Get the static IP if it ever changes:
```bash
az containerapp env show \
  --name clinten-env \
  --resource-group rg-clinten \
  --query "properties.staticIp" \
  --output tsv
```

## GitHub secrets (post-apply)

OIDC auth requires no stored credential — only identifiers:

```bash
terraform output github_actions_client_id  → AZURE_CLIENT_ID
terraform output github_actions_tenant_id  → AZURE_TENANT_ID
<subscription_id from terraform.tfvars>    → AZURE_SUBSCRIPTION_ID
```

Set on the GitHub repository (not scoped to an environment):
```
ACR_USERNAME    → Azure portal → Container Registry → Access Keys
ACR_PASSWORD    → Azure portal → Container Registry → Access Keys
```

Set on the `prod` GitHub environment:
```
SWA_DEPLOYMENT_TOKEN  → terraform output -raw creator_ui_deployment_token
```

Set on the `stage` GitHub environment:
```
SWA_DEPLOYMENT_TOKEN  → terraform output -raw creator_ui_stage_deployment_token
```

## Ongoing changes

```bash
cd infra
terraform plan    # always review before applying
terraform apply
```

OIDC federated credentials are scoped to GitHub environments (`environment:prod`, `environment:stage`). If you add a new environment or rename one, update `aad.tf` accordingly.
